# 第二阶段规划：提升 AI 准确度与交互体验

本文档聚焦第二阶段的第一优先级：**提升 AI 返回结果的准确度**，以及**优化与 AI 交互的体验**（支持可编辑 SQL、支持在第一次问题基础上继续优化/深入问题）。

适用范围：`profit_analyst_mvp` 一阶段链路（自然语言 → LLM 生成 SQL → `sql_guard` 校验 → SQLite 查询 → LLM 基于结果输出结论）。

---

## 1. 目标与非目标

### 1.1 目标

- **准确度**：LLM 生成的 SQL 更贴合用户意图、更符合库表 schema、更少出现不可执行/跑偏的查询。
- **可控性**：允许用户在界面中**修改 SQL 并运行**，但系统仍能保证只读与安全边界。
- **连续对话**：支持“在首次问题基础上继续优化/深入”，并且上下文可见、可编辑、可回滚。
- **可追溯**：每次生成/编辑/自动修复的 SQL 都有版本记录，可对比、可回滚。

### 1.2 非目标（本阶段暂不做或不优先）

- 不把系统一次性做成“全指标/全口径/全权限”的完整 BI。
- 不强依赖大规模新工程拆分；优先在一阶段项目上迭代实现。

---

## 2. 准确度优化建议（从“更懂数据”到“可纠错闭环”）

### 2.1 强约束：只在“可见 schema”内生成 SQL

- **Schema 摘要**：启动时（或定期）生成一份摘要，至少包含：
  - 表名、字段名、字段类型
  - 主键/常用连接键（若将来允许多表）
  - 常用枚举值（如站点、国家、币种等）
- **生成约束**：LLM 生成 SQL 时必须引用 schema 中存在的表/字段；不在 schema 的标识符视为错误并触发重写。

落地形态建议：
- 现有 `schema.py` 的信息扩展为“给 LLM 的 schema 摘要”与“给校验器的 schema 清单”两份输出（同源、不同表达）。

### 2.2 两段式生成：先“意图→查询计划”，再“计划→SQL”

将一次生成拆成两步可显著降低跑偏：

1) **查询计划（可读、可校验）**：包含但不限于
- 目标指标（例如利润/利润率/订单数）
- 维度（例如国家/站点/日期）
- 过滤条件（时间范围、站点、国家、品类等）
- 聚合口径（按日/周/月；是否去重；分母定义）
- 默认值（例如默认近 30 天、默认 Top 20）

2) **SQL 生成**：只允许把“计划”翻译成 SQL，不允许自行引入新口径。

### 2.3 执行前静态校验：拦截低级错误与危险语句

在 `sql_guard` 的基础上建议增加（或更严格执行）：

- **标识符校验**：表/列必须存在；禁止歧义列；必要时要求显式列名。
- **语义校验**：`GROUP BY` 与聚合列匹配、`ORDER BY` 引用列合法等。
- **安全护栏（只读）**：
  - 只允许 `SELECT`（含 `WITH`）与单语句
  - 禁止 `INSERT/UPDATE/DELETE/DROP/ALTER` 等写操作
  - 默认加 `LIMIT`（避免意外全表扫描）
  - 默认加时间范围（当用户未指定时）
  - 查询超时/扫描行数阈值（执行层约束）

> 安全提示：即使允许用户编辑 SQL，执行层也应保持只读约束与单语句限制，避免 SQL 注入与误操作风险。

### 2.4 执行后闭环纠错：基于数据库报错自动修复 SQL

将数据库错误信息纳入“自动纠错闭环”：

- **失败即修复**：捕获错误（缺列、类型不匹配、语法错等）→ 回喂 LLM → 要求“只改 SQL，不改需求/计划”→ 生成新版本。
- **成功但可疑**：对结果做轻量 sanity check（例如空结果、行数异常暴涨、金额全为 0、出现大量负值等）：
  - 给出提示（可能原因与下一步建议）
  - 或自动生成“更合理的替代查询版本”（例如补默认时间范围/补过滤条件/补去重口径）

### 2.5 指标/口径字典固定化（减少“同词不同义”）

建立“指标字典/口径字典”，使 LLM 生成更稳定：

- 为常用指标提供**确定义**（利润、毛利率、GMV、订单数、退款率等）。
- 生成 SQL 时强制引用字典定义，避免随意变更分子/分母或字段组合。

### 2.6 金标问题集回归（防止越改越偏）

用少量高频问题做“自动回归”：

- 选 20～50 个高频问题
- 固化期望 SQL 模式（或期望结果范围/行数范围/关键字段包含）
- 每次调整提示词/校验/执行策略后自动跑一遍，确保稳定性

## 3. 交互体验优化建议（让用户能改、能追问、能回滚）

### 3.1 把一次对话变成“可编辑工作区”

推荐页面结构（同屏）：

- **用户意图**：自然语言输入 + 可选的结构化参数（时间范围/站点等）
- **AI 生成 SQL（可编辑）**：提供编辑器、格式化、运行按钮
- **结果区**：表格 + 简要结论 +（可选）图表

关键点：
- 用户修改 SQL 后可“一键运行”
- 系统记录本次变更（用于追溯与回滚）

### 3.2 上下文机制（可控、可视、可编辑）

为了支持“在第一次问题基础上继续优化/深入”：

- 每轮保存结构化上下文：时间范围、站点、过滤条件、口径、SQL 版本号等
- 页面展示“当前上下文摘要”，用户可点选修改（而不是反复口头描述）

### 3.3 三类追问快捷入口（降低表达成本）

提供三类按钮/快捷操作：

- **继续优化**：在不变口径下优化过滤/维度/排序（如“按店铺拆分”“只看 Top 20 SKU”）
- **深入原因（drill-down）**：整体 → 国家/站点 → 品类 → SKU →（可选）按天/订单明细
- **对比分析**：近 30 天 vs 前 30 天、A 店 vs B 店、活动前 vs 活动后

这些操作应尽量转换为“结构化参数变更 + SQL 自动改写/重生成”，而不是纯文本解释。

### 3.4 SQL 版本树与回滚（支持迭代优化）

每次以下动作都产生一个版本节点：

- AI 生成
- 用户编辑
- 系统自动修复（基于报错/可疑结果）

能力：
- 版本列表/树形展示
- 对比 diff（至少显示“本次改了什么”）
- 一键回滚到任意版本

### 3.5 解释要“可操作”，不是长段文字

AI 输出解释建议采用结构化要点，并可点击编辑：

- 使用了哪些表/字段
- 过滤条件（时间范围、站点等）
- 聚合口径与排序方式
- 限制条数（LIMIT）与默认值

每个要点都能触发 UI 控件修改，并自动反映到 SQL 中。

### 3.6 允许编辑 SQL 的安全策略（硬边界）

即使产品层支持“修改 SQL”，执行层仍建议强制：

- **只读账号/只读连接**（强烈建议）
- **单语句执行**（禁止多语句；禁止分号串联）
- **禁止写操作**（DDL/DML）
- **限制资源**：超时、最大返回行数、最大扫描量（按 SQLite 能力或应用层限制）
- 对所有“用户输入值”（日期、站点名等）优先采用参数化绑定（若执行层支持）

---

## 4. 最小可行落地（M2.1：低复杂度高收益的 5 件事）

建议按顺序落地，尽快产生可感知的提升：

1. **Schema 摘要 + 指标/口径字典**接入到每次生成
2. **SQL 静态校验 + 只读安全护栏**（默认 `LIMIT`/默认时间范围/禁写/单语句）
3. **执行错误回喂 → 自动修复 SQL**（基于报错重写并生成新版本）
4. **SQL 编辑器 + 版本历史 + 一键回滚**
5. **上下文摘要面板 + 继续优化/深入/对比三类追问入口**

---

## 5. 验收口径（建议）

- **可执行率**：LLM 首次生成 SQL 的可执行比例提升；失败能自动修复并在 1～2 次内成功。
- **跑偏率**：用户反馈“与问题不一致”的比例下降；上下文可视化减少误解。
- **交互效率**：同一问题从“提出 → 得到满意答案”的平均轮次下降；追问更容易。
- **安全性**：任何路径均无法执行写操作；恶意/误操作 SQL 被拦截并给出明确提示。
