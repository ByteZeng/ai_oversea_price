# Agent 编程指南（Cursor 用）

> 项目：AI 数据分析助手（第一阶段）  
> 目标：以**最小实现**跑通「自然语言问题 → 生成 SQL → 查询数据 → 中文结论」的端到端链路。  
> 设计哲学：**单体优先、链路优先、约束驱动、两段式 LLM**。严禁为“未来扩展”增加不必要层次。

---

## 1. 交付范围与非目标

### 1.1 本阶段必须做到

- Streamlit 页面可启动并交互
- 使用 DeepSeek（OpenAI 兼容）调用 LLM
- LLM #1 生成可执行 SQL（允许偶发失败，但可重试/一次修正后成功）
- SQLite 返回查询结果（Mock 数据）
- LLM #2 基于结果表输出**不与表格显著冲突**的中文结论
- 页面展示：问题、SQL、结果表、结论、错误信息（可读）

### 1.2 明确不做

- 不接真实业务库、不做 ETL
- 不引入网关、队列、异步任务、缓存、权限审计体系
- 不追求“最优 SQL / 科研级因果推断”

---

## 2. 推荐技术栈（与文档一致）

- **语言**：Python（建议 3.10+）
- **UI**：Streamlit
- **数据**：SQLite（本地 `.db` 或内存）+ 启动时初始化
- **LLM**：DeepSeek API（OpenAI 兼容 SDK/客户端）
- **表格展示**：pandas（推荐）

### 2.1 环境变量（必须可配置）

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`（如需）
- `DEEPSEEK_MODEL`（SQL 生成模型）
- `DEEPSEEK_MODEL_ANALYSIS`（结论生成模型，可与上者相同）
- 可选：`APP_DB_PATH`、`DEFAULT_DAYS_WINDOW`

---

## 3. 目录与文件约定（建议）

> 可根据现有仓库结构适配，但需保持“可读、少文件、少层级”。

- `app.py`：Streamlit 入口（或 `src/app.py`，但入口必须清晰）
- `src/`
  - `db.py`：SQLite 连接、初始化、执行查询
  - `schema.py`：白名单表/字段、指标定义（revenue/profit/margin）
  - `sql_guard.py`：SQL 校验与拦截（只允许 SELECT 等）
  - `llm.py`：DeepSeek 客户端封装（超时、重试、模型选择）
  - `prompts.py`：两段 Prompt 文本（schema + 规则 + 输出格式约束）
  - `orchestrator.py`：编排（问题→SQL→校验→执行→截断/聚合→结论）
- `docs/`：现有文档保留
- `requirements.txt`：依赖清单
- `README.md`：启动方式与环境变量说明
- `seed.sql` 或 `seed.py`：建表 + 插入 Mock 数据（任选其一，保持简单）

---

## 4. 数据与 Schema 规则（强约束）

### 4.1 表清单（单表）

- 仅允许一张表：`yibai_oversea.dashbord_new_data1`（订单事实/订单粒度）
- 本阶段**不允许 `JOIN`**（单表查询），以降低 LLM 生成错误率。

### 4.2 字段来源

- 不在代码里随意新增业务字段名；字段白名单必须来源于你们已确认的《数据库字段中英文对照表》。
- 若对照表缺字段类型/可空/默认值，先落在补充清单中，再更新 schema。

### 4.3 派生指标（不落表，只在 SQL 里算）

- `订单毛利率 = 订单毛利润 / 销售额`
- `订单净利率 = 订单净利润 / 销售额`

---

## 5. SQL 生成与执行安全（必须遵守，防注入）

> 用户规则：**禁止产生 SQL 注入漏洞**。

### 5.1 允许的 SQL 子集（LLM #1 输出必须满足）

- **只允许 `SELECT`**（可选允许 `WITH`；若不稳定就禁用）
- `FROM` 仅允许 `yibai_oversea.dashbord_new_data1`
- 禁止 `JOIN`
- 字段只能来自白名单（表名.字段名）
- 聚合优先（按 `country`、按日/周），避免大量明细
- 默认时间对比：近 N 天 vs 前 N 天（N 固定或配置）

### 5.2 必须拦截的危险关键字/语句

执行前对 SQL 做硬拦截（大小写不敏感，必要时正则）：

- DDL/DML：`DROP`、`DELETE`、`INSERT`、`UPDATE`、`ALTER`、`CREATE`、`REPLACE`
- 外部/系统：`ATTACH`、`DETACH`、`PRAGMA`、`VACUUM`
- 多语句：禁止 `;` 拼接多条（如出现 `;`，直接拒绝或只取第一条并再次校验）

### 5.3 参数化与字符串拼接

- 若后续引入筛选条件（时间窗、国家等），必须使用**参数化查询**（SQLite `?` 占位符），禁止将用户输入直接拼到 SQL 字符串中。
- LLM 生成的 SQL 若包含用户文本（如国家名），需要改造为参数化或做严格枚举映射。

### 5.4 结果行数与入模限制

- 查询结果过大：在送入 LLM #2 前必须**截断或先聚合**（例如限制行数、只送汇总表）。
- 结论 Prompt 明确要求：**不得引入表外事实**，所有结论必须引用结果表的对比/趋势。

---

## 6. LLM 两段式流程（实现准则）

### 6.1 LLM #1（SQL 生成）

- 输入：Schema/字段解释/指标定义 + 硬规则 + 用户问题
- 输出：仅 SQL（不夹带解释文字，不要 Markdown code fence）
- 失败策略：解析失败/校验失败时，可做一次“自我修正”重试；仍失败则向 UI 输出可读错误

### 6.2 LLM #2（结论生成）

- 输入：用户问题 + 最终执行 SQL + 小型结果表（聚合/截断后）
- 输出：中文结论，必须包含至少 1–2 个来自表格的定量/方向性引用（例如“近期美国站 margin 下降，shipping_cost 均值上升”）
- 禁止：编造不存在的维度/字段/数值

---

## 7. Streamlit UI（最小可用）

页面至少包含：

- 输入框：自然语言问题（单行或多行均可）
- 按钮：开始分析
- 四段输出区：
  - 原始问题
  - 生成 SQL（可复制）
  - 查询结果表格（dataframe）
  - 中文结论
- 错误展示：API 失败、SQL 校验失败、SQL 执行失败、空结果（原因要清楚）
- 可选：展示当前模型名、默认时间窗 N（便于排障，不显示敏感信息）

---

## 8. 开发顺序（严格按链路推进）

按里程碑推进（不要跨级加戏）：

- **M1 可运行骨架**：Streamlit 可启动，无 LLM 也能展示页面
- **M2 数据就绪**：SQLite 初始化 + 手工 SQL 可跑出汇总结果
- **M3 LLM SQL 通路**：能生成并执行可用 SQL（带校验与拦截）
- **M4 结论通路**：基于结果表输出中文解释（不冲突）
- **M5 演示级交付**：示例问题端到端跑通，README/截图/说明补齐

粗估总工时 6–10 小时；超时优先排查是否做了范围外工作。

---

## 9. 验收用例（必须通过）

- **TC-01**：输入示例问题并点击分析，不出现未捕获异常
- **TC-02**：页面展示的 SQL 为 `SELECT`，且只涉及白名单表字段
- **TC-03**：表格有数据；若无数据，提示原因明确
- **TC-04**：结论提及表格中的关键对比（国家/时间段/利润或利润率方向），且不与表格显著冲突
- **TC-05（可选）**：断网或错误 Key 时，展示可读错误信息

---

## 10. Cursor 编程执行规则（给后续自动编程用）

- 优先改动少、文件少、层级浅；每次改动都要能运行
- 任何与 SQL 相关的逻辑，必须先过 `sql_guard`（只允许子集 + 危险拦截）
- 任何来自用户/LLM 的字符串都视为不可信输入；需要白名单/枚举/参数化
- 结论生成只喂“聚合/截断后的结果表”，避免 token 成本与胡编
- 输出错误要可读、可定位（提示是 LLM、校验、执行、还是空结果）
