# 需求规格说明书（SRS）

**项目名称**：AI 数据分析助手（第一阶段）  
**文档版本**：v0.1  
**状态**：草案

---

## 1. 背景与问题陈述

业务人员需要快速获得对「利润率/利润下降」的**可解释初步结论**。第一阶段不追求完备分析平台，而是验证一条可复用链路：

> 自然语言问题 → 受约束的 SQL → 查询结果 → 自然语言结论

---

## 2. 目标（本阶段唯一目标）

交付一个**本地可运行**的最小系统，能够回答：

- **示例问题**：「美国站最近利润率下降原因」

并满足闭环输出：**SQL（可不完美）→ 数据（真实或 Mock）→ 分析结论**。

---

## 3. 范围界定

### 3.1 范围内（必须交付）

| ID | 需求简述 | 优先级 |
| --- | --- | --- |
| FR-01 | 用户输入一个自然语言问题（单场景：利润下降/利润率相关） | P0 |
| FR-02 | 系统调用 LLM 生成 SQL（仅允许使用既定表与字段白名单） | P0 |
| FR-03 | 系统执行 SQL 并返回结果表（初期允许 Mock / 本地 SQLite 固定数据集） | P0 |
| FR-04 | 系统将「问题 + SQL + 结果表」交给 LLM 生成分析结论文本 | P0 |
| FR-05 | 界面展示：输入区、触发分析、SQL 文本、结果表、结论文本 | P0 |

### 3.2 范围外（本阶段明确禁止）

以下项**不得**纳入第一版交付，以避免范围膨胀：

- 用户系统（注册/登录/权限）
- 多问题类型路由（例如同时支持库存、投放、退款等）
- 动态 Schema 发现与自适应建表
- Agent 框架、多工具编排、复杂工作流引擎
- 微服务拆分、过度解耦、为“未来扩展”而增加抽象层

> 说明：以上为**管理性约束**，与是否“技术上可行”无关；违反即视为偏离阶段目标。

---

## 4. 术语与指标定义（第一版需写死）

为避免“利润/利润率”语义漂移，第一阶段在实现中必须**固定**（与《数据与 Schema 设计》一致）：

- **收入（revenue）**：与 `price` 字段口径一致（或定义为 `price`）
- **成本项**：至少包含订单 `cost`，以及按 `order_id` 关联的 `shipping_cost`
- **利润（profit）**：`price - cost - shipping_cost`（字段名与 join 规则以 Schema 文档为准）
- **利润率（margin）**：`profit / NULLIF(revenue, 0)`（需处理除零）

“最近”若无用户指定，采用系统默认时间窗（例如近 14 天对比前 14 天），并在 UI 或结果区**显式展示**，避免误解。

---

## 5. 用户故事（单角色）

**作为**业务分析者，  
**我希望**用自然语言提问并看到 SQL、数据与结论，  
**以便**快速定位利润率变化的主要贡献因素（哪怕是粗粒度）。

**验收口径**：见《开发计划与验收标准》。

---

## 6. 非功能需求（NFR）

| ID | 类别 | 要求 |
| --- | --- | --- |
| NFR-01 | 运行形态 | 以本机启动的 Web 界面为主（默认单用户、本机访问） |
| NFR-02 | 机密性 | API Key 等密钥不得硬编码入库；不入公共仓库 |
| NFR-03 | 可观测性（最低） | 页面可展示 SQL 与错误信息，便于调试 Prompt/SQL |
| NFR-04 | 性能 | 第一阶段以“请求在可接受时间内返回”为目标，不做性能优化 KPI |
| NFR-05 | 数据安全（最低） | 默认仅本地 SQLite/Mock；若后续接真实库需另开阶段评估 |

---

## 7. 假设与依赖

1. 使用 **DeepSeek** 等云端大模型 API（OpenAI 兼容调用方式）。  
2. 开发机器可访问外网 API；若不可用需切换为离线方案（不在第一版范围内，除非阻塞交付）。  
3. 第一阶段数据以 **Mock/SQLite** 为主，不对齐真实业务库字段全集。
