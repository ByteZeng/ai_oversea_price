# 数据与 Schema 设计

**项目名称**：AI 数据分析助手（第一阶段）  
**文档版本**：v0.1  
**状态**：草案

---

## 1. 设计目标

1. **字段总量控制**：尽量控制业务字段数量，避免 Prompt 过长导致不稳定（具体以《数据库字段中英文对照表》与后续《字段补充清单》为准）。
2. **指标可解释**：指标口径与字段解释以对照表为准；文档中**不写死具体字段名**，避免实现与数据变更时频繁返工。

---

## 2. 表定义（逻辑模型）

本章节**不自行定义任何业务字段**。字段名（中英文）、解释与口径，必须以你已确认的《数据库字段中英文对照表》为准。

### 2.1 表清单（仅表名与关系）

- `yibai_oversea.dashbord_new_data1`：订单事实表（订单粒度）

### 2.2 字段来源与落地方式

- **字段来源**：请将你已确认的《数据库字段中英文对照表》粘贴/维护到仓库（建议放 `docs/数据库字段中英文对照表.xlsx` 或 `docs/数据库字段中英文对照表.md`）。  
- **Schema 生成规则**：仅从该对照表生成：
  - 表名、字段名（英文）
  - 字段中文名/解释（用于提示词与报表展示）
  - 字段口径（例如“price 是否含税/是否含运费”）
  - 字段类型与可空（若对照表未给出，则进入“待补充清单”）

### 2.3 待你补充的信息（Excel 清单）

仓库将提供一个可用 Excel 打开的清单文件：`docs/字段补充清单.csv`。  
当对照表中缺少以下任一项时，请在该清单里补齐（我会再据此更新 schema 定义）：

- 字段所属表
- 英文字段名
- 中文字段名/解释（如对照表已有可不填）
- 数据类型（SQLite/MySQL/Postgres 等具体类型）
- 是否可空、默认值
- 单位/币种/是否含税
- 示例值（1-3 个）

---

## 3. 派生指标（不在表中存，允许在 SQL 中计算）

在分析中允许计算“利润/利润率/毛利/净利”等派生指标，但**具体使用哪些字段**、字段口径（是否含税/运费等）必须以《数据库字段中英文对照表》为准；当对照表不完整时，以 `docs/字段补充清单.csv` 补齐后再进入提示词与实现。

> 若 Mock 数据需要体现“利润率下降”，优先从对照表定义的“成本/费用类字段上升”或“结构变化（站点/时间/渠道/品类等）”制造可解释差异。

---

## 4. SQL 生成约束（写进 Prompt 的硬规则示例）

1. 只允许 `SELECT`（以及必要的 `WITH`，若团队允许；否则第一版可禁用 `WITH` 以降低复杂度）。
2. `FROM` 仅允许 `yibai_oversea.dashbord_new_data1`（单表查询，不允许 `JOIN`）。
3. 字段只能来自白名单。
4. 时间对比：默认近 N 天 vs 前 N 天（N 在配置或 UI 上固定）。
5. 聚合优先：按 `country`、按周/按日聚合，避免返回海量明细入模。

---

## 5. Mock 数据策略（建议）

- **行数**：每个时间桶几十到几百行即可。  
- **美国站故事线（选一即可）**：
  - 近期 `shipping_cost` 均值显著高于前期；或  
  - 近期 `cost` 上升；或  
  - 近期低价订单占比提高导致 `margin` 被稀释。

将“故事线”写入数据生成脚本注释，避免团队忘记 Mock 的可解释目的。

---

## 6. 数据初始化方式

1. 应用启动：连接 SQLite → `CREATE TABLE IF NOT EXISTS` → `INSERT` 种子数据（或从 `seed.sql` 执行）。
2. 第一阶段**不**做 ETL、不接入生产库。

---

## 7. 隐私与安全

Mock 数据不得包含真实客户 PII；若后续接真实数据，需另立数据分级与访问控制文档（不在 v0.1）。
