# 开发计划与验收标准

**项目名称**：AI 数据分析助手（第一阶段）  
**文档版本**：v0.1  
**状态**：草案

---

## 1. 里程碑（建议）

| 里程碑 | 产出 | 完成定义 |
| --- | --- | --- |
| M0 | 文档定稿（本套） | SRS/TDD/数据/验收对齐无矛盾 |
| M1 | 可运行骨架 | Streamlit 可启动；无 LLM 也能展示页面 |
| M2 | 数据就绪 | SQLite 初始化成功；手工 SQL 能跑出汇总结果 |
| M3 | LLM SQL 通路 | 能生成可执行 SQL 并返回结果（允许偶发失败但可重试成功） |
| M4 | 结论通路 | 第二段 Prompt 能输出与表格一致的中文解释 |
| M5 | 演示级交付 | 走完示例问题端到端；文档更新最终截图/说明 |

**粗估工时**：6–10 小时（与阶段目标一致）；超时优先检查是否做了范围外工作。

---

## 2. 任务分解（WBS）

### 2.1 工程与环境

- Python 版本确认（建议 3.10+）  
- 依赖清单：`streamlit`、`openai`（或兼容客户端）、`pandas`（可选但推荐用于表格展示）  
- 环境变量：`DEEPSEEK_API_KEY` 等

### 2.2 数据层

- `seed.sql` 或 `seed.py`：建表 + 插入 Mock  
- 手工验证：至少 2 条 SQL（全站汇总、美国站分桶对比）

### 2.3 LLM 层

- Prompt v1：SQL 生成（schema + 规则）  
- Prompt v2：结果解释（强约束“只基于表格”）  
- 最小防护：SQL 危险关键字拦截、结果行数限制

### 2.4 UI 层

- 输入/按钮/四段展示（问题、SQL、表、结论）  
- 错误展示：API 失败、SQL 失败、空结果

## 3. 测试策略（第一阶段可接受的最小集合）

### 3.1 手工测试用例（必须通过）

| 编号 | 步骤 | 期望 |
| --- | --- | --- |
| TC-01 | 输入示例问题并点击分析 | 不出现未捕获异常 |
| TC-02 | 检查页面展示的 SQL | 为 `SELECT`，且只涉及白名单表字段 |
| TC-03 | 检查表格 | 有数据（或明确提示无数据原因） |
| TC-04 | 检查结论文本 | 提及表格中的关键对比（国家/时间段/利润或利润率方向） |
| TC-05 | 断网或错误 Key（可选） | 有可读错误信息 |

### 3.2 自动化（可选，不强制）

- 对 SQL 校验函数做单元测试（若实现关键字拦截）  
- 对指标计算做 1–2 个固定样例测试（若抽出纯函数）

## 4. 验收标准（与 SRS 对齐）

必须同时满足：

1. 用户可输入自然语言问题（至少覆盖“利润/利润率下降原因”表述）。  
2. 系统可生成 SQL（允许不完美，但应可执行或可被一次修正后执行）。  
3. 系统可返回查询结果（Mock/SQLite）。  
4. 系统可输出中文分析结论，并与查询结果不显著冲突。

**完成即视为阶段成功**：不追加“必须最优 SQL/必须因果推断正确”等过高门槛。

## 5. 风险清单与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| LLM 频繁生成非法 SQL | 阻塞演示 | 收紧 Prompt；减少 SQL 复杂度；模板化 SQL |
| 结论编造表外信息 | 误导 | Prompt 强约束 + 只喂聚合表 |
| API 不稳定 | 体验差 | timeout + 有限重试；错误提示 |
