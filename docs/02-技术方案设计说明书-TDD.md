# 技术方案设计说明书（TDD）

**项目名称**：AI 数据分析助手（第一阶段）  
**文档版本**：v0.1  
**状态**：草案

---

## 1. 设计原则（对齐阶段约束）

1. **单体优先**：单进程应用完成 UI、编排、数据库访问与 LLM 调用。  
2. **链路优先**：优先跑通「问题 → SQL → 数据 → 结论」，不为扩展性增加层次。  
3. **约束驱动**：通过字段白名单与 SQL 允许集合降低 LLM 失效率。  
4. **两段式 LLM**：SQL 生成与结果解释拆分，降低胡编概率。

---

## 2. 推荐技术栈

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 语言 | Python（团队最熟） | 与数据分析生态一致，便于 SQLite/pandas |
| 界面 | Streamlit | 本机快速交付表单与表格展示 |
| LLM | DeepSeek API | 成本低；采用 OpenAI 兼容客户端/SDK |
| 数据 | SQLite（`:memory:` 或本地 `.db`）+ 固定初始化 SQL | 第一阶段不接真实业务库 |
| 配置 | 环境变量（`.env` 可选） | `DEEPSEEK_API_KEY`、`DEEPSEEK_BASE_URL`、`DEEPSEEK_MODEL` 等 |

---

## 3. 总体架构（逻辑视图）

```
┌─────────────┐    ┌──────────────────┐    ┌─────────────┐    ┌──────────────────┐
│ Streamlit UI│ -> │ Orchestrator      │ -> │ SQLite      │ -> │ Analysis Prompt  │
│ 输入问题     │    │ SQL 生成(LLM #1)  │    │ 执行 SQL     │    │ 结论生成(LLM #2)   │
└─────────────┘    │ 校验/执行/重试     │    └─────────────┘    └──────────────────┘
                   └──────────────────┘
```

**说明**：不引入消息队列、网关、独立 AI 服务进程。

---

## 4. 核心模块职责

### 4.1 UI（Streamlit）

- 输入：用户问题（单行或多行文本框其一即可）  
- 操作：开始分析按钮  
- 输出：SQL 文本、查询结果表格（`dataframe`）、结论文本  
- 可选：展示默认时间窗、模型名（便于排障，不含敏感信息）

### 4.2 SQL 生成（LLM #1）

- **输入**：固定 Schema 说明 + 指标定义 + 用户问题 + SQL 风格约束（只允许 `SELECT`，必须白名单字段等）  
- **输出**：仅 SQL 文本  
- **失败策略（第一版保持简单）**：
  - 解析失败 → 提示用户重试或返回错误信息  
  - 可选：单次“自我修正”再请求（仍属同一条链路，不引入 Agent）

### 4.3 SQL 执行

- 使用 SQLite 参数化查询（若未来需要）；第一版若以“模型生成字符串”为主，至少执行前做**危险关键字拦截**（`DROP/DELETE/INSERT/UPDATE/ATTACH/PRAGMA` 等）  
- 结果行数过多时：**在入模分析前先做聚合或截断**（产品规则，不是架构优化）

### 4.4 分析结论（LLM #2）

- **输入**：用户问题 + SQL + 小结果表（推荐仅聚合结果）  
- **输出**：中文结论，要求引用结果表中的趋势/对比（“第一版可用”口径，不追求科研级因果）

---

## 5. 与 DeepSeek 的集成要点

1. 使用 OpenAI 兼容方式设置 `base_url` 与 `api_key`。  
2. **SQL 生成**可选用更偏代码/推理的模型；**结论**可用 chat 模型（具体以控制台可用模型为准）。  
3. 请求设置合理 `timeout`；对 429/5xx 采用有限重试（实现阶段决定，设计层先记录需求）。

---

## 6. 日志与调试（最小）

- 在页面展示 SQL 与错误文本即满足“可调试最低线”。  
- 不在第一版引入复杂日志平台；可选保存最后一次请求到本地文件（若做，需注意脱敏）。

---

## 7. 风险设计应对（工程化但不过度）

| 风险 | 对策 |
| --- | --- |
| SQL 经常写错 | 更强 schema 约束；限制 join 路径；限定时间函数；必要时提供“模板 SQL + LLM 填空”降级 |
| 结论胡编 | 强制只基于结果表；结果表必须是聚合级别；提示中写清“不得引入表外事实” |
| Token 成本 | 第二段只喂聚合结果；限制行数 |

---
